# Technical Specification: Messaging & Notifications (F-007)

**Last Updated:** Tue Jan 06 2026 by Tech Lead (Addressed spec review feedback)
**Status:** 🟡 In Dev

## 1. Overview

Send **Gate Pass** to visitors via WhatsApp when their visit is approved. The gate pass is a composite pamphlet image already generated by `SecurityService.approveVisit()`.

### Design Philosophy
- **Simple & Synchronous:** No message queues. Direct API calls with error handling.
- **Extend, Don't Rebuild:** Add `WhatsAppService` to existing `MessagingModule`.
- **Graceful Degradation:** WhatsApp failures must NOT block the approval flow.

### Provider
**Meta WhatsApp Business API** (Official Cloud API) - Direct integration.

#### Branch Lookup Logic (in `NotificationsService.notifyVisitorOnApproval`)

```typescript
// Inject PrismaService in NotificationsService constructor
constructor(
  private readonly prisma: PrismaService,
  private readonly whatsAppService: WhatsAppService,
) {}

async notifyVisitorOnApproval(visit: Visit, visitor: Visitor): Promise<void> {
  // Guard: No phone number
  if (!visitor.phone) {
    this.logger.log(`Visitor ${visitor.id} has no phone, skipping WhatsApp notification`);
    return;
  }

  // Fetch branch name for template variable
  const branch = await this.prisma.branch.findUnique({
    where: { id: visit.branchId },
    select: { name: true },
  });

  if (!branch) {
    this.logger.warn(`Branch ${visit.branchId} not found for visit ${visit.id}`);
    return;
  }

  // Send gate pass via WhatsApp
  await this.whatsAppService.sendGatePass(
    visitor.phone,
    branch.name,
    visit.visitQRCode, // base64 pamphlet image
  );
}
```

### 4.4 Phone Number Normalization

> **WhatsApp API requires phone numbers in E.164 format WITHOUT the `+` prefix.**
> Example: `919876543210` (not `+919876543210`)

**Add to `WhatsAppService`:**

```typescript
/**
 * Normalize phone number to WhatsApp format (E.164 without +)
 * @param phone - Input in various formats
 * @returns Normalized phone (e.g., "919876543210") or null if invalid
 *
 * Handles:
 * - "+919876543210" → "919876543210"
 * - "919876543210"  → "919876543210"
 * - "9876543210"    → "919876543210" (assumes India +91)
 * - "09876543210"   → "919876543210" (removes leading 0)
 */
private normalizePhone(phone: string): string | null {
  // Remove all non-digit characters
  let normalized = phone.replace(/\D/g, '');

  // Handle leading 0 (domestic format)
  if (normalized.startsWith('0')) {
    normalized = normalized.substring(1);
  }

  // If 10 digits, assume India (+91)
  if (normalized.length === 10) {
    normalized = '91' + normalized;
  }

  // Validate: should be 12 digits for India (91 + 10 digits)
  if (normalized.length < 10 || normalized.length > 15) {
    this.logger.warn(`Invalid phone number format: ${phone}`);
    return null;
  }

  return normalized;
}
```

**Usage in `sendGatePass`:**
```typescript
async sendGatePass(to: string, branchName: string, gatePassBase64: string): Promise<boolean> {
  const phone = this.normalizePhone(to);
  if (!phone) {
    return false;
  }
  // ... rest of implementation
}
```

### 4.5 Media Handling: Upload-First Approach

> **WhatsApp API supports two media methods: URL or Media ID. We use Media ID (upload-first).**

**Media Constraints (per Meta docs):**
| Type | Format | Max Size | Notes |
|------|--------|----------|-------|
| PNG  | `image/png` | **5 MB** | 8-bit, RGB or RGBA only |
| JPEG | `image/jpeg` | 5 MB | 8-bit, RGB or RGBA only |

> **IMPORTANT:** Validate image size before upload. Gate pass images exceeding 5 MB will be rejected by Meta's API.

**Flow:**
1. Upload image binary to Meta's Media endpoint → receive `media_id`
2. Send template message referencing `media_id` in header

**Why Media ID over URL:**
- No public hosting required (avoids GCP public bucket)
- Media cached on WhatsApp's CDN
- More reliable delivery

**Media Upload Endpoint:**
```
POST https://graph.facebook.com/v24.0/{PHONE_NUMBER_ID}/media
Content-Type: multipart/form-data

- file: <binary image data>
- type: image/png
- messaging_product: whatsapp
```

> **Note:** Uploaded media IDs expire after 30 days. This is not a concern for our use case (immediate send after upload).

**Response:**
```json
{ "id": "1234567890" }  // This is the media_id
```

**Template Message with Media ID:**
```json
{
  "messaging_product": "whatsapp",
  "to": "919876543210",
  "type": "template",
  "template": {
    "name": "gate_pass_approved",
    "language": { "code": "en" },
    "components": [
      {
        "type": "header",
        "parameters": [
          { "type": "image", "image": { "id": "1234567890" } }
        ]
      },
      {
        "type": "body",
        "parameters": [
          {
            "type": "text",
            "text": "City Hospital - Main Branch",
            "parameter_name": "hospital_branch"
          }
        ]
      }
    ]
  }
}
```

### Common Meta API Errors

| Code     | Meaning              | Action              |
| -------- | -------------------- | ------------------- |
| `131047` | User hasn't opted in | Skip silently       |
| `131053` | MIME type mismatch   | Verify image is valid PNG |
| `132000` | Template not found   | Check template name |
| `132015` | Param count mismatch | Verify 1 body param |

## 8. Edge Cases

| Case                             | Handling                         |
| -------------------------------- | -------------------------------- |
| Missing WhatsApp config          | Log warning, return `false`      |
| 10-digit phone (no country code) | Prepend `91` (India)             |
| **Image exceeds 5 MB**           | Log error, return `false` (Meta limit) |
| Media upload fails               | Log error, return `false`        |
| Visitor has no phone             | Log info, skip notification      |
| Invalid base64 image             | Log error, return `false`        |

## Production Issue & Resolution

**Date:** Tue Jan 06 2026

### Issue: Error Code 100 - "Parameter name is missing or empty"

**Error Message:**
```
ERROR [WhatsAppService] Failed to send WhatsApp message: 400 - 
{"error":{"message":"(#100) Invalid parameter","type":"OAuthException","code":100,
"error_data":{"messaging_product":"whatsapp","details":"Parameter name is missing or empty"}}}
```

**Root Cause:**
The WhatsApp template in Meta Business Suite was configured with **named variables** (e.g., `{{hospital_branch}}`), which requires either:
1. Adding `parameter_name` attribute to the payload parameters, OR
2. Reconfiguring the template to use positional variables (e.g., `{{1}}`)

**Resolution:**
The WhatsApp template in Meta Business Suite uses **named variables** (`{{hospital_branch}}`), which requires the `parameter_name` attribute in the payload. The implementation was updated to include:

```typescript
{
  type: 'text',
  text: branchName,
  parameter_name: 'hospital_branch',
}
```

**Code Updates:**
- Added `parameter_name: 'hospital_branch'` to body parameters
- Added error code 100 handling with clear error messages
- Cleaned up debug logging code
- Updated test expectations to match new payload structure
- Updated `MetaApiError` interface to include `error_data` field

**Status:** ✅ Resolved - Implementation updated to match Meta template configuration
