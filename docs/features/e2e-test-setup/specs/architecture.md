# E2E Testing Architecture - Playwright Setup

## 1. Overview
This document outlines the architecture for End-to-End (E2E) testing of the Hospital Visitor Tracking System using Playwright. 

### Goals
- Provide a stable, deterministic environment for automated UI testing.
- Eliminate dependencies on external messaging services (AWS SNS, Meta WhatsApp API) during tests.
- Simplify authentication for test automation.
- Ensure consistent data across test runs.

### Non-Goals
- Testing real SMS delivery (handled via unit/integration tests).
- Load testing or performance testing.

## 2. Environment Configuration

The system uses environment variables to toggle test behaviors. These should be defined in `backend/.env` for local testing.

| Variable | Description | Recommended Value |
|----------|-------------|-------------------|
| `TEST_MODE` | Enables test-specific logic in backend | `true` |
| `E2E_FIXED_OTP` | The OTP used for all logins in test mode | `123456` |

## 3. Backend Modifications

### 3.1 AuthService: Fixed OTP Logic
When `TEST_MODE` is enabled, the `AuthService` will use a hardcoded OTP instead of generating a random one. This allows Playwright to perform logins without reading logs or intercepting network requests.

**Logic Update (Pseudocode):**
```typescript
// Proposed change in backend/src/auth/auth.service.ts
async login(loginDto: LoginDto) {
  const isTestMode = this.configService.get('TEST_MODE') === 'true';
  const otp = isTestMode 
    ? this.configService.get('E2E_FIXED_OTP') 
    : Math.floor(100000 + Math.random() * 900000).toString();
  
  // Store hashed OTP in DB ...
  
  if (!isTestMode) {
    await this.smsService.sendOtp(loginDto.phone, otp);
  } else {
    this.logger.log(`[TEST_MODE] OTP for ${loginDto.phone}: ${otp}`);
  }
}
```

### 3.2 Messaging Services: Mocking Behavior
To avoid external API calls and potential costs/throttling, `SmsService` and `WhatsAppService` will bypass their respective providers when `TEST_MODE` is active.

- **SmsService**: Logs the message to console but skips `SNSClient.send()`.
- **WhatsAppService**: Logs the intent and returns `true` (success) without calling Meta Graph API.

## 7. Security Considerations

### 7.1 Production Guard Pattern
All test-mode logic MUST be wrapped in a double-check pattern:

```typescript
// Safe Pattern - Use in ALL services that have TEST_MODE logic
private isTestModeEnabled(): boolean {
  const isTestMode = this.configService.get('TEST_MODE') === 'true';
  const isProduction = process.env.NODE_ENV === 'production';
  
  // NEVER enable test mode in production
  if (isProduction && isTestMode) {
    this.logger.error('SECURITY: TEST_MODE cannot be enabled in production!');
    return false;
  }
  
  return isTestMode;
}
```

### 7.2 Environment Matrix

| NODE_ENV | TEST_MODE | Result |
|----------|-----------|--------|
| `development` | `true` | ✅ Test mode enabled |
| `development` | `false` | Standard behavior |
| `production` | `true` | ❌ **BLOCKED** - Logs error, uses standard behavior |
| `production` | `false` | Standard behavior |

### 7.3 CI/CD Considerations

> ⚠️ **WARNING**: Ensure your CI/CD pipeline sets `NODE_ENV=test` (not `production`) when running E2E tests. Never deploy with `TEST_MODE=true`.

Recommended CI configuration:
```yaml
env:
  NODE_ENV: test
  TEST_MODE: true
  E2E_FIXED_OTP: "123456"
```
