## 5. Orchestration Flow

### 5.1 Main Method: `registerVisitor()`

The main method follows this orchestration sequence:

1. **Validate Branch Access:** Ensure `dto.branchId` matches `user.branchId`
2. **Visitor Lookup:** Check if visitor exists by phone + branch (for reuse logic)
3. **Upload Files to GCP:** Upload photo and optional documents BEFORE transaction (returns URLs)
4. **Execute Transaction:** Atomically perform visitor upsert and visit creation with CHECKED_IN status
5. **Generate Gate Pass:** Create OTP, generate gate pass image, upload to GCP
6. **Parallel Operations:** Execute `updateVisitWithGatePass()` and `sendWhatsAppNotification()` in parallel using `Promise.allSettled()`
7. **Handle WhatsApp Failure:** Log warning if WhatsApp fails (non-blocking, does not affect success response)
8. **Build Response:** Fetch complete visitor and visit data, return with WhatsApp status

### 5.2 Transaction Flow

The Prisma transaction (`executeRegistrationTransaction()`) performs:

1. **Visitor Upsert:** Create new visitor or update existing visitor by `phone + branchId` composite key
   - Update all fields (name, email, company, photo, documents)
   - Set `phoneVerified: true`
2. **Visit Creation:** Create visit record with:
   - `status: 'CHECKED_IN'` (skip approval workflow)
   - `checkInTime: now()`
   - `checkedInById: user.id`
   - Empty OTP fields (populated after gate pass generation)
   - Visit type-specific fields (MEETING: department, host; DELIVERY: platform, recipient)

## 6. Transaction Handling Approach

### 6.1 Transaction Scope

- **Visitor Upsert:** Create new or update existing by phone + branch
- **Visit Creation:** Create visit with CHECKED_IN status

### 6.2 Transaction Atomicity

- **Success:** Both visitor and visit committed atomically
- **Failure:** Both rolled back, no partial state
- **Files Trade-off:** Uploaded BEFORE transaction; orphaned GCP files acceptable if transaction fails

### 6.3 Error Handling

```typescript
try {
```

## 7. Error Handling

| Error Code | HTTP Status | Scenario |
|------------|--------------|----------|
| `BRANCH_MISMATCH` | 400 | Request branchId does not match user branchId |
| `FILE_UPLOAD_ERROR` | 400 | GCP file upload failed |
| `VISITOR_ALREADY_EXISTS` | 409 | Phone + branch unique constraint violation |
| `GCP_UPLOAD_FAILED` | 500 | Gate pass upload to GCP failed |
| `GATE_PASS_GENERATION_FAILED` | 500 | Gate pass image generation failed |
| `WHATSAPP_SEND_FAILED` | 200 (warning) | WhatsApp delivery failed (non-blocking) |

### WhatsApp Non-Blocking Failure

- **Strategy:** Use `Promise.allSettled()` for parallel GCP update + WhatsApp send
- **Logging:** Log warning with visit ID and error reason
- **Response:** Set `whatsappSent: false`, return success
- **Frontend:** Display warning message but proceed with success flow

### 8.1 Unit Test Cases

- **Happy Path - Meeting Visit (New Visitor):**
  - Meeting registration with all required fields
  - Photo and optional government ID document upload
  - New visitor profile created, visit in CHECKED_IN status
  - Gate pass generated and uploaded to GCP
  - WhatsApp sent successfully

  - Visit created for same visitor
  - TEST_MODE returns fixed OTP "654321"

- **WhatsApp Failure (Non-Blocking):**
  - Registration completes successfully
  - WhatsApp API throws error during send
  - Response returns `success: true`, `whatsappSent: false`
  - Warning logged with visit ID

- **File Upload Error:**
  - GCP upload throws error
  - `BadRequestException` thrown with `FILE_UPLOAD_ERROR` code
  - No database changes made

- **Branch Mismatch Validation:**
  - Request branchId differs from user branchId
  - `BadRequestException` thrown with `BRANCH_MISMATCH` code
  - No further processing occurs

- **Gate Pass Generation Failure:**
  - Gate pass service throws error during image generation or upload
  - `InternalServerErrorException` thrown
  - Transaction may have already committed (data exists but no gate pass)

## 10. Notes

- **TEST_MODE Support:** GatePassService checks `TEST_MODE` environment variable and returns fixed OTP `"654321"` when enabled
- **Orphaned Files:** If transaction fails after file upload, files remain in GCP (acceptable trade-off for simplicity)
- **Visitor Profile Reuse:** Upsert ensures existing visitor profiles are updated with new information if phone matches
- **Direct Check-In:** Security personnel verify identity in person, so approval workflow is bypassed
- **Audit Trail:** `checkedInById` and `checkInTime` fields track who and when check-in occurred
