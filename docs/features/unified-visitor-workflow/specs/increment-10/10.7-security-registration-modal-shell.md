## 1. Overview

Create `SecurityRegistrationModal` component as a container for security-assisted visitor registration flow. The modal handles multi-step state management (visit type selection → registration form → success), displays as a bottom sheet on mobile (< 768px) with drag handle, and as a centered modal on desktop. Includes type selection step with Meeting and Delivery cards (emerald/amber theming), focus trap, ESC-to-close, and proper ARIA attributes.

### Key Requirements
- Container component for security-assisted registration workflow
- Local state machine with steps: `visitType` → `visitorForm` → `success`
- Responsive: bottom sheet (mobile) with drag handle, centered modal (desktop)
- Type selection with Meeting (emerald) and Delivery (amber) cards
- Focus trap and ESC key to close
- ARIA `role="dialog"` and `aria-modal="true"`
- Pre-filled phone from PhoneLookupFlow
- Open/close animations (slide-up/fade)
- Back navigation with unsaved changes confirmation

## 6. Logic Flow (Summary)

### 6.1 Component Initialization
- Initialize state with `step: 'visitType'`, empty visitor data
- Create refs for modal focus management
- Detect mobile vs desktop using media query

### 6.2 Modal Lifecycle
- **On Open:** Save currently focused element, focus first interactive element in modal after 100ms
- **On Close:** Restore focus to previously focused element, reset all state to initial values

### 6.3 Keyboard Events
- **ESC Key:** Trigger `handleCloseWithConfirmation()` when modal is open
- **Tab / Shift + Tab:** Navigate within focus trap (see Accessibility section)

### 6.4 Step Rendering
- Switch on `state.step` to render appropriate component
- `visitType`: Render `VisitTypeSelectionStep` with Meeting/Delivery cards
- `visitorForm`: Placeholder for Task 10.8 forms
- `success`: Render confirmation with visitor data

### 6.5 Navigation Handlers
- **Visit Type Selection:** Update state to `step: 'visitorForm'` with selected type, clear previous form data
- **Back Navigation:** Check for unsaved changes, show confirmation if data exists, otherwise return to type selection
- **Close with Confirmation:** Check for unsaved changes, show confirmation dialog if data exists, otherwise call `onClose()`

## 8. Accessibility (Consolidated)

### ARIA Attributes
- **Modal Container:** `role="dialog"`, `aria-modal="true"`, `aria-labelledby="modal-title"`
- **Modal Title (screen reader only):** Dynamic based on step:
  - `visitType`: "New Visitor Registration"
  - `visitorForm`: "Meeting Registration" / "Delivery Registration"
  - `success`: "Registration Successful"
- **Description (type selection step):** `aria-describedby="modal-description"` with "Select visit type to continue with visitor registration"

### Focus Management
- **On Mount:** Focus first interactive element in modal
- **On Unmount:** Restore focus to element that triggered modal
- **Focus Trap:** Implement trap to keep Tab/Shift+Tab within modal boundaries
  - Query all focusable elements: buttons, links, inputs, select, textarea, [tabindex]
  - Wrap from last to first element on Tab, first to last on Shift+Tab
  - Prevent default on boundary crosses

### Keyboard Navigation
- **ESC:** Close modal (with confirmation if unsaved changes)
- **Tab:** Navigate forward through focusable elements
- **Shift + Tab:** Navigate backward
- **Enter:** Activate selected type card, submit form, action buttons

### Screen Reader Support
- Modal title and description announced on mount
- Step changes announced via live region (optional enhancement)
- Unsaved changes confirmation dialog announced
- Success state announced with visitor name

## 12. Implementation Notes

### Progressive Enhancement
This task implements the container shell and type selection step. The actual form rendering for visitorForm step will be completed in Task 10.8 (Meeting registration) and Task 10.9 (Delivery registration). Include a placeholder rendering for the form steps until those tasks are complete.

### State Management Approach
Use local state with `useState` because:
- Modal is isolated to single component
- No need for global context
- Short-lived state (cleared on close)
- Follows existing patterns in codebase

### Unsaved Changes Detection
The `hasUnsavedChanges()` function checks:
- Form data exists (not null)
- Step is `visitorForm`
- Form has any non-empty required fields

### Drag Gesture Implementation
For mobile bottom sheet drag handle:
- Use touch events (`touchstart`, `touchmove`, `touchend`)
- Track Y position changes
- Close modal if drag exceeds 50% of modal height
- Or use existing Radix UI Dialog/Sheet primitive if available
