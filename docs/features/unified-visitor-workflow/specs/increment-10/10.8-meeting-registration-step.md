## 1. Overview

Implement the meeting registration step within `SecurityRegistrationModal` (from Task 10.7). This step renders a form that collects all required meeting visitor fields. The key differences from the public flow are:

- Phone is pre-filled and read-only (from PhoneLookupFlow)
- Step indicator: "Step 2 of 2 • Meeting Registration"
- Submit button text: "Register & Check In"
- No navigation to previous step (back returns to type selection with confirmation)

### Key Requirements

- Reuse existing `MeetingDetailsStep` component for host/department selection
- Phone field: pre-filled from props, read-only, gray background
- Collect: firstName*, lastName*, email*, company, designation, address, photo*, governmentIdDocument*, officeIdDocument (optional), host*, department, purpose
- Form validation: on blur + on submit (schema from Task 10.4)
- Back navigation: confirmation if form has unsaved data
- Submit: calls `registerSecurityVisitor()` API (from Task 10.4)

## 6. Logic Flow

### 6.1 Form Submission Handler

```typescript
const handleMeetingSubmit = useCallback(async (formData: MeetingFormData) => {
  setState((prev) => ({ ...prev, isLoading: true, error: null }));

  try {
    // Prepare DTO
    const requestDto: SecurityMeetingRegistrationDto = {
      phone: formData.phone,
      branchId: branchId,
      firstName: formData.firstName,
      lastName: formData.lastName,
      email: formData.email,
      company: formData.company || null,
      designation: formData.designation || null,
      address: formData.address || null,
      visitType: 'MEETING',
      department: formData.department,
      hostSelectionMode: formData.hostSelectionMode,
      hostId: formData.hostId || null,
      staffName: formData.staffName || null,
      staffPhone: formData.staffPhone || null,
      purpose: formData.purpose,
    };

    // Call API (from Task 10.4)
    const response = await registerSecurityVisitor(requestDto, {
      photo: formData.photo!,
      governmentIdDocument: formData.governmentIdDocument!,
      officeIdDocument: formData.officeIdDocument || null,
    });

    // Transition to success step
    setState((prev) => ({
      ...prev,
      step: 'success',
      visitorData: formData,
      registrationResponse: response,
      isLoading: false,
    }));

  } catch (error) {
    setState((prev) => ({
      ...prev,
      isLoading: false,
      error: error.response?.data?.message || 'Registration failed. Please try again.',
    }));
  }
}, [branchId]);
```

### 6.2 Back Navigation Handler

```typescript
const handleBackNavigation = useCallback(() => {
  const formData = form.getValues();
  const hasData = formData.firstName || formData.lastName || formData.email ||
                   formData.company || formData.designation || formData.address ||
                   formData.photo || formData.governmentIdDocument || formData.officeIdDocument ||
                   formData.department || formData.hostId || formData.staffName ||
                   formData.staffPhone || formData.purpose;

  if (hasData) {
    setShowUnsavedChangesDialog(true);
  } else {
    setState((prev) => ({ ...prev, step: 'visitType', visitType: null, visitorData: null }));
    form.reset();
  }
}, [form]);
```

## 8. Styling & Accessibility

**Styling Reference:** See `docs/features/unified-visitor-workflow/UX-DESIGN-SECURITY-REGISTRATION.md`

**Key Style Points:**
- Step header: text-lg font-semibold, text-sm muted-foreground subtitle

- Photo capture: 120px x 120px minimum button
- Document upload: dashed border with centered upload button

**Accessibility:**
- Form: `role="form"` and `aria-labelledby="form-heading"`
- Required fields: `*` indicator and `aria-required="true"`
- Error messages: linked via `aria-describedby`
- Tab order: First Name → Last Name → Email → Company → Designation → Address → Photo → Government ID → Office ID → Department → Host → Purpose → Back → Cancel → Submit
- Enter key triggers submission on last field
- Focus: first field on mount, first invalid on error
- Screen reader: step indicator announced, read-only phone announces "disabled, pre-filled", errors via `role="alert"`
