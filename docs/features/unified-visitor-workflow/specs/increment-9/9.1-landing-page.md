# Task 9.1: Landing Page Specification

## 1. Task Description

Create the public-facing landing page at `/visitor-registration` that serves as the entry point for the unified visitor registration workflow. This page displays a welcoming interface with the hospital icon, dynamically fetched hospital name (from branchId), welcome message, illustration, and a primary "Start Registration" button that navigates to the multi-step registration wizard.

## 2. Component/Location

**File Path:** `frontend/src/app/visitor-registration/page.tsx`

**Component Name:** `VisitorRegistrationLandingPage`

**Page Type:** Server Component (Next.js App Router)

## 7. Accessibility

### 7.1 Heading Hierarchy

**Proper Semantic Structure:**
```html
<main>
  <h1>Welcome to [Hospital Name]</h1> <!-- Main page title, h1 -->
  <h2>Visitor Registration</h2>     <!-- Section label, h2 -->

```

**Requirements:**
- Single `<h1>` per page
- Logical heading order (h1 → h2)
- No skipped heading levels

### 7.2 ARIA Labels

**Decorative Elements:**
- Hospital icon: `aria-hidden="true"` (purely decorative icon)
- Illustration: `aria-hidden="true"` (purely decorative)

**Interactive Elements:**
- Button: Use built-in button semantics
- Link: Use built-in link semantics (if using `<Link>`)

### 7.3 Keyboard Navigation

**Tab Order:**
- Default tab order follows DOM structure
- Focusable elements:
  1. "Start Registration" button (only focusable interactive element)

**Focus Management:**
- Default: Focus on first focusable element on page load
- No custom focus trapping needed (simple page)

**Focus Styles:**
- Button: Uses built-in `focus-visible` styles from Button component
- Links: Use `ring-ring/50` or similar focus indicator

### 7.4 Screen Reader Support

**Semantic HTML:**
- Use `<main>` wrapper for main content
- Use `<h1>` for main heading
- Use `<h2>` for secondary heading
- Use `<button>` for interactive elements

**Live Regions:**
- Not required (static page)

**Dynamic Content:**
- Hospital name: Announce when loaded via `aria-live="polite"` if fetching from backend
- If fetch fails, announce generic welcome message

**Skip Links (Optional):**
- Consider adding "Skip to main content" link if part of larger layout

### 7.5 WCAG AA Contrast Compliance

**Text Contrast:**
- Normal text: Minimum 4.5:1 contrast ratio
- Large text (18px+): Minimum 3:1 contrast ratio
- Interactive elements: Minimum 3:1 contrast ratio

**Verification:**
- Use Tailwind color system (designed for accessibility)
- Test with browser developer tools or axe DevTools
- Primary button (emerald-500) passes contrast with white text

### 7.6 Color Independence

**Requirements:**
- Do not rely solely on color to convey information
- Use icons, text, and layout to supplement color

**Implementation:**
- Illustration uses icon + color
- Button uses text + icon + color
- Labels use text alone

## 9. Error Handling

### 9.1 Error Cases

**Branch Name Resolution Errors:**

1. **Missing branchId Parameter:**
    - URL missing `?branchId=` query parameter
    - Display generic welcome: "Welcome to Hospital"
    - Log warning to console
    - Allow user to proceed with registration

2. **Invalid branchId:**
    - Backend returns 404 for GET /branches/:branchId
    - Display generic welcome: "Welcome to Hospital"
    - Show subtle error message: "Unable to load hospital details"
    - Allow user to proceed with registration

3. **Backend API Failure:**
    - Network error when fetching branch name
    - Display generic welcome: "Welcome to Hospital"
    - Show error message with retry option
    - Log error for debugging

**Navigation Errors:**

4. **Navigation Failure:**
    - Next.js Link handles most navigation errors
    - Show error boundary if route doesn't exist (Task 9.2 not yet implemented)

### 9.2 Branch Name Resolution Logic

**Branch ID Extraction:**
```typescript
// Extract branchId from URL searchParams
// Example URL: /visitor-registration?branchId=abc123

interface SearchParams {
  branchId?: string;
}

// Server Component approach:
const searchParams = await props.searchParams;
const branchId = searchParams?.branchId;
```

**Branch Name Fetch:**
```typescript
// Fetch branch name using pre-existing endpoint
// GET /branches/:branchId (from IMPLEMENTATION-REPORT.md Section 2.3)

interface BranchResponse {
  id: string;
  name: string;
  // ... other branch fields
}

// Fetch logic:
// 1. If branchId exists, fetch branch details from backend
// 2. If fetch succeeds, display branch name in welcome message
// 3. If fetch fails (404, 500, network error), display generic "Hospital"
// 4. If branchId missing, display generic "Hospital"
```

**Welcome Message Variants:**
- **Success:** "Welcome to {Branch Name}"
- **Fallback:** "Welcome to Hospital"
- **Loading:** "Welcome to Hospital" (with loading indicator if client-side fetch)

**Implementation Notes:**
- Use pre-existing `GET /branches` endpoint for branch lookup
- Branch model from FEATURE-ARCHITECTURE.md Section 2.1 contains `name` field
- Fetch can be done on server (Server Component) or client (useEffect with TanStack Query)
- Recommended: Server Component fetch for better performance and SEO

### 9.3 Loading States

**Conditional Loading States:**

**Server Component Approach (Recommended):**
- No client-side loading state needed
- Page renders with branch name fetched during server-side rendering
- Fallback to "Welcome to Hospital" if fetch takes too long

**Client Component Approach:**
- Initial render: Show "Welcome to Hospital" (fallback)
- Fetch branch name in useEffect
- Update welcome message once data arrives
- Optional: Show subtle loading indicator next to hospital name

**No Blocking Loading State:**
- Page should remain interactive during branch name fetch
- "Start Registration" button should be clickable regardless of branch name fetch status

### 9.4 Empty States

**Not Applicable:**
- Page always displays content (welcome message, illustration, button)
- No conditional rendering based on data
- Branch name fetch failures use fallback text rather than empty state
