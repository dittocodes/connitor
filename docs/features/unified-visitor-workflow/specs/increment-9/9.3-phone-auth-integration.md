# Phone Authentication Integration

## 1. Task Description

Wire `PhoneEntryStep` and `PhoneVerificationStep` components into the unified visitor registration wizard to establish phone authentication as the entry point for all public visitor registrations. This integration enables phone number verification via SMS OTP before proceeding to visit type selection and subsequent registration steps.

### Validation Rules
- **Phone Number**: Exactly 10 digits, numeric only
- **Branch ID**: Valid UUID v4 format
- **OTP**: Exactly 6 digits, numeric only
- **OTP Expiry**: 5 minutes from generation time
- **OTP Attempts**: Maximum 3 failed attempts before lockout

## 4. API Integration

### POST /public/visitors/send-otp

**Request Body**:
```typescript
interface SendOtpRequest {
  phone: string;      // 10 digits
  branchId: string;   // UUID v4
}
```

**Success Response** (201):
```typescript
interface SendOtpResponse {
  success: true;
  message: string;           // "A 6-digit code has been sent to your mobile via SMS."
  isNewVisitor: boolean;     // Indicates first-time visitor at branch
  testOtp?: string;          // Included only when TEST_MODE=true
}
```

**Error Response Types**:
```typescript
interface SendOtpErrorResponse {
  success: false;
  code: 'VALIDATION_FAILED' | 'RATE_LIMIT_EXCEEDED' | 'SMS_SEND_FAILED' | 'OTP_LOCKED';
  message: string;
  retryAfter?: number;  // Seconds until retry allowed (for rate limit)
}

interface VerifyPhoneErrorResponse {
  success: false;
  code: 'INVALID_OTP' | 'OTP_EXPIRED' | 'OTP_LOCKED' | 'VISITOR_NOT_FOUND';
  message: string;
  attemptsRemaining?: number;  // Remaining OTP attempts
}

interface NetworkErrorResponse {
  code: 'NETWORK_ERROR';
  message: string;
  retryable: true;
}
```

**Error Responses**:
- `400` Bad Request: Returns `SendOtpErrorResponse` or `VerifyPhoneErrorResponse` (validation failed, OTP locked, SMS send failed, visitor not found)
- `429` Too Many Requests: Returns `SendOtpErrorResponse` with `code: 'RATE_LIMIT_EXCEEDED'`, `retryAfter` field (3 requests/IP/hour)

### POST /public/visitors/verify-phone

**Request Body**:
```typescript
interface VerifyPhoneRequest {
  phone: string;      // 10 digits
  otp: string;        // 6 digits
  branchId: string;   // UUID v4
}
```

**Success Response** (200):
```typescript
interface VerifyPhoneResponse {
  verified: true;
  isExistingVisitor: boolean;
  visitorData: {
    id: string;
    firstName: string;
    middleName: string | null;
    lastName: string;
    phone: string;
    email: string | null;
    company: string | null;
    designation: string | null;
    phoneVerified: boolean;
  };
}
```

**Error Responses**:
- `400` Bad Request: Invalid OTP, expired OTP, locked (3 attempts)
- `404` Not Found: Visitor not found

### API Call Timing
1. **Send OTP**: Called when user clicks "Send OTP" on PhoneEntryStep
2. **Verify OTP**: Called when user completes 6-digit OTP entry on PhoneVerificationStep

### Step Navigation Logic

**PHONE_ENTRY → PHONE_VERIFICATION**:
- Read branchId from URL query parameter: `new URLSearchParams(window.location.search).get('branchId')`
- User enters 10-digit phone number and clicks "Send OTP"
- Wizard calls `POST /public/visitors/send-otp`
- On success: Update `phoneData` with phone, branchId; Transition to PHONE_VERIFICATION
- On error: Display inline error; Stay on PHONE_ENTRY

**PHONE_VERIFICATION → VISIT_TYPE_SELECTION**:
- User enters 6-digit OTP
- Wizard calls `POST /public/visitors/verify-phone`
- On success: Update `phoneData` with verification result, visitorId, isExistingVisitor; Store existing visitor data for pre-fill; Transition to VISIT_TYPE_SELECTION
- On error: Display error; Show attempts remaining; Stay on PHONE_VERIFICATION
- On 3 failed attempts: Lock phone entry for 15 minutes; Show error with retry time

**Back Navigation**:
- PHONE_VERIFICATION back button → Return to PHONE_ENTRY (clears OTP state)
- Previous steps have no back navigation

**Cancel Flow**:
- Both steps have Cancel button (X in header or Cancel link)
- On cancel: Reset wizard state to initial; Return to landing page

```typescript
<PhoneVerificationStep
  phoneData={wizardState.phoneData}
  onNext={async (otp) => {
    const result = await api.verifyPhone({
      phone: phoneData.phone,
      otp,
      branchId: phoneData.branchId
    });
    updateWizardState({
      phoneData: { ...phoneData, isVerified: true, visitorId: result.visitorData.id }

    navigateTo(WizardStep.VISIT_TYPE_SELECTION);

  onBack={() => navigateTo(WizardStep.PHONE_ENTRY)}
  onCancel={handleCancel}
  isLoading={isVerifyingOtp}
  error={verifyOtpError}
/>
```

## 6. State Management

### Phone Data Structure
```typescript
interface PhoneDataState {
  phone: string;
  branchId: string;
  isVerified: boolean;
  visitorId?: string;
  isNewVisitor?: boolean;
  otpExpiry?: string;
  attemptsRemaining?: number;
  verifiedAt?: string;
}
```

### State Transitions
1. **Initial State**: `phoneData = null`
2. **After Send OTP**: `phoneData = { phone, branchId, isVerified: false, isNewVisitor }`, `otpExpiry = now + 5 minutes`
3. **After Verify OTP (Success)**: `phoneData.isVerified = true`, `phoneData.visitorId = result.visitorData.id`, `phoneData.verifiedAt = now`
4. **After Verify OTP (Failed)**: `phoneData.attemptsRemaining = 3 - failedAttempts`; If `attemptsRemaining === 0`: Lock entry for 15 minutes

### Existing Visitor Data Pre-fill

When `verifyPhone` returns `isExistingVisitor: true`:
1. Store `visitorData` in wizard state for subsequent steps
2. **Visit Type Selection**: Skip or auto-select based on visit history (optional enhancement)
3. **Registration Forms**: Pre-fill firstName, middleName, lastName, email, company, designation; Phone already pre-filled from Step 1a; Photo and ID documents still required (security requirement)

```typescript
function prefillRegistrationData(visitorData: ExistingVisitorData): Partial<RegistrationData> {
  return {
    visitor: {
      firstName: visitorData.firstName,
      middleName: visitorData.middleName ?? undefined,
      lastName: visitorData.lastName,
      email: visitorData.email ?? undefined,
      company: visitorData.company ?? undefined,
      designation: visitorData.designation ?? undefined,
    },
  };
}
```

## 7. Navigation Flow

### Step Transition Rules

| From Step            | To Step                       | Trigger                         | Condition                |
|----------------------|-------------------------------|---------------------------------|--------------------------|
| PHONE_ENTRY          | PHONE_VERIFICATION            | "Send OTP" button click         | API success              |
| PHONE_VERIFICATION   | VISIT_TYPE_SELECTION          | OTP verification success        | API success              |
| PHONE_VERIFICATION   | PHONE_ENTRY                   | "Change Phone Number" link      | User action              |
| Any Step             | Landing Page                  | Cancel button/link              | User action (reset)      |

### Back Navigation Handling
- **PHONE_VERIFICATION**: Shows "← Change Phone Number" link below OTP input
- **VISIT_TYPE_SELECTION onwards**: Shows "← Back" button in header
- **PHONE_ENTRY**: No back button (entry point)

### Cancel Flow Handling
1. **Cancel Button Location**: X icon in header or Cancel link at bottom
2. **On Cancel**: Clear all wizard state; Reset step to PHONE_ENTRY; Navigate to `/visitor-registration` landing page
3. **Confirmation Not Required**: Cancel is destructive (discards progress)

## 8. Error Handling

### Error Types & User Messages

| Error Type              | HTTP Status | User Message                                      | Recovery Action                |
|-------------------------|-------------|---------------------------------------------------|--------------------------------|
| Network Error           | N/A         | "Connection lost. Please check your internet and try again." | "Retry" button re-sends last request |
| Invalid Phone Format    | 400         | "Please enter a valid 10-digit phone number."    | User corrects input            |
| Rate Limit Exceeded     | 429         | "Too many attempts. Please try again in [X] minutes." | Wait or contact security       |
| OTP Send Failed         | 400         | "Unable to send SMS. Please try again or contact security." | Retry button                   |
| Invalid OTP             | 400         | "Invalid code. Please check and try again. [X] attempts remaining." | Re-enter OTP                   |
| OTP Expired             | 400         | "Code expired. Please request a new code."       | Click "Resend OTP" link        |
| OTP Locked (3 attempts) | 400         | "Too many failed attempts. Please try again after [time]." | Wait for lockout period        |
| Visitor Not Found       | 404         | "Visitor not found. Please complete verification again." | Restart wizard from beginning  |

### Error State Handling

**Send OTP Errors**:
- Display inline error below phone input
- Highlight phone input border in red
- Preserve phone number input
- Enable "Send OTP" button for retry

**Verify OTP Errors**:
- Display inline error below OTP input
- Highlight OTP input boxes in red (`hasError` prop on OtpInput)
- Show attempts remaining countdown
- Disable "Resend OTP" during 60-second cooldown
- Enable "Resend OTP" after cooldown expires

### Network Error Recovery Pattern
```typescript
displayError({
  message: "Connection lost. Please check your internet and try again.",
  actionLabel: "Retry",
  onAction: retryLastRequest,
  persistent: true
});
```

### Rate Limit Error Handling
- Response includes `retryAfter` header (seconds until allowed)
- Display countdown timer: "Resend code in X:XX"
- Disable "Send OTP" / "Resend OTP" during rate limit window
- After timer expires: Re-enable button with label change

## 9. Accessibility

### Focus Management
**Step Transition**: Save current focused element; Transition to new step; Move focus to first focusable element (PHONE_ENTRY: phone input, PHONE_VERIFICATION: first OTP digit input)

**Back Button Press**: Restore focus to phone input field (PHONE_ENTRY)

**Cancel Action**: Move focus to "Start Registration" button on landing page

### Keyboard Navigation
- **Tab**: Logical tab order through form fields and buttons
- **Shift+Tab**: Reverse navigation
- **Enter**: Submit form when on primary button (Send OTP / Verify & Continue)
- **Escape**: Cancel wizard (optional, requires confirmation)
- **OTP Input**: Support arrow keys (left/right), backspace, paste (handled by OtpInput component)

### Screen Reader Announcements

**ARIA Live Regions**:

### Visual Accessibility
- Color contrast: WCAG AA compliant (verified with design system)
- Error states: Red border + text + icon (not color-only indication)
- Focus states: Visible focus ring on all interactive elements
