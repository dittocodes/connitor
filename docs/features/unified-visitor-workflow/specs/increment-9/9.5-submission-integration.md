# Task 9.5: Submission and Confirmation Flow Integration

## 1. Task Description

Wire the submission of visit registration data to the backend API and integrate the confirmation flow into the unified visitor registration wizard. This task handles the transition from visit details entry (Delivery or Meeting) to the ConfirmationStep upon successful submission, and provides the navigation path to status and gate pass pages.

**Key Responsibilities:**
- Call `POST /public/visitors` endpoint on VISIT_DETAILS step submission
- Prepare submission payload using `MeetingRegistrationDto` or `DeliveryRegistrationDto` based on visit type
- Handle file uploads (visitor photo, government IDs, office IDs) using FormData multipart
- Store `visitId` from submission response for subsequent navigation
- Transition wizard state to CONFIRMATION step on successful submission
- Integrate ConfirmationStep component with appropriate props and callbacks
- Handle "Done" button callback to navigate to status page

## 4. API Integration

### POST /public/visitors

#### Request Format (Delivery)
```
URL: POST /public/visitors
Content-Type: multipart/form-data

FormData: { phone, branchId, visitCategory: "DELIVERY", firstName, middleName?, lastName,
           deliveryPlatform, deliveryRecipient?, orderReference?, visitSubType, photo }
```

#### Request Format (Meeting)
```
URL: POST /public/visitors
Content-Type: multipart/form-data

FormData: { phone, branchId, visitCategory: "MEETING", firstName, middleName?, lastName,
           email, alternatePhone?, alternateEmail?, company?, companyWebsite?, designation,
           address?, department?, personToMeet?, staffName?, staffPhone?, purpose,
           reportingManagerName?, reportingManagerPhone?, visitSubType, photo,
           governmentId?, officeId? }
```

#### Response Format (Success)
```
HTTP Status: 201 Created
{ success: true, message: "Visit request submitted successfully", visitId: string,
  visitStatus: "REQUEST_SENT", visitorId: string, visitorName: string }
```

#### Error Responses
- **400 Bad Request**: `{ success: false, message: "Validation failed", errors: [{ field, message }] }`
- **401 Unauthorized**: `{ success: false, message: "Phone verification required", code: "PHONE_NOT_VERIFIED" }`
- **500 Internal Server Error**: `{ success: false, message: "Server error occurred", code: "SERVER_ERROR" }`

#### When to Call
- Triggered when user clicks "Submit" on VISIT_DETAILS step
- After all form validation passes
- After all required files are uploaded
- DO NOT call on CONFIRMATION step (it's a success display only)

### GET /public/visits/:visitId/status

#### Request Format
```
URL: GET /public/visits/:visitId/status
visitId: UUID v4 string
```

#### Response Format (200 OK)
```typescript
{
  success: true;
  data: {
    visitId: string;
    status: "REQUEST_SENT" | "APPROVED" | "REJECTED" | "CHECKED_IN" | "CHECKED_OUT";
    visitor: { id, firstName, lastName, fullName, phone, photoUrl? };
    visitCategory: "MEETING" | "DELIVERY" | null;
    submittedAt: string;
    branch: { id, name, address? };
    gatePass?: {
      checkInOtp?: string;
      validUntil?: string;
      gatePassUrl?: string;
      generatedAt?: string;
      sentViaWhatsApp?: boolean;
      isUsed?: boolean;
    };
    meetingDetails?: { purpose?, department?, staffName?, staffPhone? };
    deliveryDetails?: { platform?, recipient?, orderReference? };
  };
}
```

#### Error Responses
- **400 Bad Request**: Invalid UUID format
- **404 Not Found**: Visit not found
- **410 Gone**: Visit request has expired

### Wizard State Machine Integration

#### Step Transition Logic
```typescript
function handleVisitDetailsSubmit(formData: FormData) {
  // 1. Validate form data
  if (!validateForm(formData)) return;

  // 2. Prepare submission payload
  const payload = prepareSubmissionPayload(
    wizardState.visitType,
    wizardState.formData,
    wizardState.fileData
  );

  // 3. Call submission API
  setWizardState(prev => ({ ...prev, isSubmitting: true }));

  try {
    const response = await apiClient.post<VisitorRegistrationResponse>(
      '/public/visitors',
      payload,
      { headers: { 'Content-Type': 'multipart/form-data' } }
    );

    // 4. On success: Store visitId and transition
    setWizardState(prev => ({
      ...prev,
      visitId: response.data.visitId,
      currentStep: 'CONFIRMATION',
      submissionError: undefined
    }));

    // 5. Announce success for screen readers
    announceStatus('Visit request submitted successfully');

  } catch (error) {
    // 6. On error: Preserve form data and show error
    const submissionError = parseSubmissionError(error);
    setWizardState(prev => ({ ...prev, submissionError }));
  } finally {
    setWizardState(prev => ({ ...prev, isSubmitting: false }));
  }
}

// ConfirmationStep integration
function renderConfirmationStep() {
  return (
    <ConfirmationStep
      visitId={wizardState.visitId!}
      visitType={wizardState.visitType}
      onDone={handleDoneButtonClick}
      onContactSecurity={handleContactSecurity}
      autoRedirectDelay={null}
    />
  );
}

// Done button handler - navigate to status page
function handleDoneButtonClick() {
  if (wizardState.visitId) {
    router.push(`/visitor-registration/status/${wizardState.visitId}`);
  }
}
```

### Form Data Preparation for Submission

```typescript
function prepareSubmissionPayload(
  visitType: 'MEETING' | 'DELIVERY',
  formData: Partial<RegistrationData>,
  fileData: { visitorPhoto?: File; governmentId?: File; officeId?: File }
): FormData {
  const payload = new FormData();

  // Add base fields
  payload.append('phone', formData.phone!);
  payload.append('branchId', formData.branchId!);
  payload.append('visitCategory', visitType);
  payload.append('firstName', formData.firstName!);
  if (formData.middleName) payload.append('middleName', formData.middleName);
  payload.append('lastName', formData.lastName!);

  // Add photo
  if (fileData.visitorPhoto) {
    payload.append('photo', fileData.visitorPhoto);
  }

  // Type-specific fields
  if (visitType === 'DELIVERY') {
    const deliveryData = formData as DeliveryRegistrationData;
    payload.append('deliveryPlatform', deliveryData.deliveryPlatform);
    if (deliveryData.deliveryRecipient) {
      payload.append('deliveryRecipient', deliveryData.deliveryRecipient);
    }
    if (deliveryData.orderReference) {
      payload.append('orderReference', deliveryData.orderReference);
    }
    payload.append('visitSubType', deliveryData.visitSubType);
  } else {
    const meetingData = formData as MeetingRegistrationData;

    payload.append('email', meetingData.email);
    if (meetingData.alternatePhone) {
      payload.append('alternatePhone', meetingData.alternatePhone);
    }
    if (meetingData.alternateEmail) {
      payload.append('alternateEmail', meetingData.alternateEmail);
    }
    if (meetingData.company) payload.append('company', meetingData.company);
    if (meetingData.companyWebsite) {
      payload.append('companyWebsite', meetingData.companyWebsite);
    }
    payload.append('designation', meetingData.designation);
    if (meetingData.address) payload.append('address', meetingData.address);

    if (meetingData.department) {
      payload.append('department', meetingData.department);
    }
    if (meetingData.personToMeet) {
      payload.append('personToMeet', meetingData.personToMeet);
    }
    if (meetingData.staffName) payload.append('staffName', meetingData.staffName);
    if (meetingData.staffPhone) payload.append('staffPhone', meetingData.staffPhone);
    payload.append('purpose', meetingData.purpose);
    if (meetingData.reportingManagerName) {
      payload.append('reportingManagerName', meetingData.reportingManagerName);
    }
    if (meetingData.reportingManagerPhone) {
      payload.append('reportingManagerPhone', meetingData.reportingManagerPhone);
    }
    payload.append('visitSubType', meetingData.visitSubType);

    // Government ID (required for meetings)
    if (fileData.governmentId) {
      payload.append('governmentId', fileData.governmentId);
    }
    if (fileData.officeId) {
      payload.append('officeId', fileData.officeId);
    }
  }

  return payload;
}
```

## 7. Navigation Flow

### Step Transition Sequence
1. User completes VISIT_DETAILS step (DeliveryDetailsStep or MeetingDetailsStep)
2. User clicks "Submit" button
3. Wizard validates form data
4. Wizard calls POST /public/visitors
5. On success: Store visitId; Set currentStep to 'CONFIRMATION'; Clear errors
6. Wizard renders ConfirmationStep with visitId and visitType
7. User views success message and "Done" button
8. User clicks "Done" button
9. Wizard navigates to /visitor-registration/status/:visitId
10. Status page polls for approval
11. On approval, status page redirects to /visitor-registration/gate-pass/:visitId

### Flow Summary
```
VISIT_DETAILS (Submit) → Validate → POST /public/visitors
  → Success: Store visitId → CONFIRMATION → (Done) → Status Page → (Approval) → Gate Pass
  → Error: Show message, keep data, allow retry
```

### Cancel Flow Handling
ConfirmationStep supports optional cancel (Escape key): No explicit cancel in design; Only "Done" and optional "Contact Security" actions; Parent wizard can handle by resetting to start or navigating elsewhere

## 8. Error Handling

### Network Error Handling
Preserve all form data in wizard state; Display "Connection lost. Please check your internet and try again."; Show "Retry" button; Keep user on VISIT_DETAILS step; Do NOT transition to CONFIRMATION

```typescript
if (isNetworkError(error)) {
  setWizardState(prev => ({
    ...prev,
    submissionError: {
      code: 'NETWORK_ERROR',
      message: 'Connection lost. Please check your internet and try again.',
      retryable: true
    }
  }));
}
```

### Submission API Error Handling

#### 400 Validation Errors
Display field-specific validation errors; Highlight invalid fields (red border); Show error icon/message next to each field; Keep user on VISIT_DETAILS step; Preserve form data for correction

#### 401 Phone Not Verified
Redirect user to PHONE_VERIFICATION step; Display error banner: "Phone verification required"; Re-initiate phone verification flow; Preserve visitor data for auto-fill after re-verification

#### 500 Server Error
Display generic error message; Show "Retry" button; Preserve form data; Log error for debugging; Keep user on VISIT_DETAILS step

### File Upload Error Handling
Validate file size/type before submission; Show inline error under file input; Allow user to select different file; Do not submit until files are valid; Preserve valid files, allow re-upload of invalid ones

**Validation Rules**: Visitor photo: Max 5MB, JPG/PNG only; Government ID: Max 5MB, PDF/JPG/PNG; Office ID: Max 5MB, PDF/JPG/PNG (optional)

## Summary

This specification defines the integration of submission and confirmation flow into the unified visitor registration wizard. Key components:
1. **Submission Handler**: Calls `POST /public/visitors` with FormData payload
2. **State Management**: Stores `visitId` on success, preserves form data on error
3. **Step Transition**: Moves from VISIT_DETAILS to CONFIRMATION on successful submission
4. **ConfirmationStep Integration**: Displays success message with "Done" button callback
5. **Navigation Flow**: "Done" button → Status page → (polls for approval) → Gate Pass page
6. **Error Handling**: Network, validation, file upload, and server errors with retry options
7. **Accessibility**: Focus management, screen reader announcements, ARIA live regions
8. **Testing**: Comprehensive unit and E2E tests covering happy paths and error scenarios

**Critical Constraints**:
- Use NEW `POST /public/visitors` endpoint (NOT pre-existing registration endpoints)
- Use NEW DTOs: `MeetingRegistrationDto` for meetings, `DeliveryRegistrationDto` for deliveries
- Handle file uploads using FormData multipart encoding
- Preserve form data on submission error for retry
- DO NOT call submission endpoint on CONFIRMATION step (submission happens on VISIT_DETAILS)
